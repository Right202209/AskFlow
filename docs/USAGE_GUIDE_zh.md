# AskFlow 系统使用流程指南

本文档旨在介绍 AskFlow 系统的端到端完整使用流程，帮助开发者、管理员和终端用户理解如何从零开始运行并使用本系统。

## 1. 系统初始化与启动

在开始使用系统前，需要先启动相关服务并进行初始化：

1. **配置环境与拉取依赖**
   按照 `README_zh.md` 的指引，安装 Python 和 Node.js 依赖 (`make install`, `make install-web`)。
2. **启动基础中间件与服务**
   运行 `make docker-up`，启动 PostgreSQL (关系型数据库)、Redis (缓存与队列)、ChromaDB (向量数据库) 以及 MinIO (对象存储)。
3. **数据库迁移与测试数据**
   - 运行 `make migrate` 执行 Alembic 数据库表结构初始化。
   - (可选) 使用 `scripts/seed_data.py` 和 `scripts/create_user.py` 创建测试账单、初始化管理员账号及基础测试配置。
4. **启动前后端控制台**
   - 后端服务：`make dev` (启动 FastAPI)
   - 前端服务：`make dev-web` (启动 Vite/React)

---

## 2. 知识库构建 (Admin / 管理员视角)

为了让 AskFlow 的 Agent 具备回答业务问题的能力，管理员需要持续维护知识库：

1. **登录管理后台**
   使用初始化的管理员账号 (Admin) 登录前端 Web 系统。
2. **文档上传与存储**
   在“文档管理”模块，上传业务 PDF/Markdown/TXT 等文档。文档原始文件会被直接存储记录入 MinIO 对象存储中。
3. **内容解析与向量化 (Embedding)**
   - 后端 `embedding` 模块中的工作流 (Chunker & Parser) 会自动解析文档文本。
   - 文本被切分为适当的 Chunk，并调用 Embedding 模型转化为向量。
   - 最终存入 ChromaDB，并使用 BM25 构建倒排索引，完成混合检索 (Hybrid Search) 的准备工作。
4. **意图配置**
   在后台调整用户的“指令/意图分类”提示词和规则，确保准确识别用户是想“查询知识”、“查询账单”还是“转人工”。

---

## 3. 智能问答与意图路由 (User / 终端用户视角)

当普通用户进入 AskFlow 的前端聊天界面后，其对话请求将经历以下智能处理流程：

1. **发起对话**
   用户在聊天框输入问题（支持文本，底层为 WebSocket 双向通信）。
2. **意图识别 (Intent Classifier)**
   后端 Agent 图 (Agent Graph) 首先拦截用户请求，调用大模型分析意图。支持的意图包括：
   - `rag` (知识库问答)
   - `tool` (工具调用，例如查订单/查账单)
   - `clarify` (意图不明确，反问用户)
   - `ticket` / `handoff` (人工客服/工单)
3. **RAG 检索与回答 (如果被路由到 `rag`)**
   - 系统去 ChromaDB + BM25 中混合检索相关 Chunk。
   - 提取最佳上下文并传给大模型。
   - 大模型流式 (Streaming Token) 输出最终解答给前端，并在末尾附上引用的文档来源。
4. **业务工具调用 (如果被路由到 `tool`)**
   系统识别到业务相关（如“帮我查一下订单号 1234”），会调用对应的 Python 工具函数查询数据库并返回结果。

---

## 4. 人工客服与工单流转 (Support Agent 视角)

AI 无法解决所有问题，AskFlow 提供了完整的退路机制：

1. **触发转人工**
   当用户主动要求“转人工”，或连续 N 次 AI 回答不佳 (被评价为踩/解决失败) 时，Agent 会路由到 `ticket` 逻辑。
2. **创建工单**
   系统自动提取聊天上下文和用户诉求，在 `tickets` 模块创建一个状态为“待处理”的工单。
3. **客服介入处理**
   - 拥有 `Agent/Admin` 身份的客服人员可以在后台“工单看板”中看到新分配或待领取的工单。
   - 客服可以直接查看先前的 AI 对话记录，快速掌握背景。
   - 客服回复工单（通过回复跟进消息或直接与用户会话），状态变更为“处理中”。
4. **工单关闭**
   问题解决后，客服或用户可将工单标记为“已解决/已关闭”。

---

## 5. 运营分析与系统优化 (Admin 视角)

长期的系统运营离不开数据支撑：

1. **看板总览 (Dashboard)**
   后台提供数据监控，包括当日会话数、成功拦截率（AI 解决未转人工的比例）、高频访问问题等。
2. **反馈回收**
   针对用户在对应消息上点的“赞/踩”，分析并优化对应文档源或业务工具提示词。
3. **迭代优化**
   管理员可以随时删除旧文档、重构文档树、上传新文档，然后触发“Reindex (重建索引)”，系统将重新切片并计算向量，实时生效。

---

## 总结

AskFlow 的核心链路可以概括为：**“数据入库 -> 用户提问 -> AI 意图判定 -> 知识/工具结合回答 -> 解决失败则落入人工工单 -> 数据分析复盘反馈”**的完整闭环智能客服生态。
